# 관계형 데이터베이스의 숙명, 느린 검색

**관계형 데이터베이스(RDB)**는 데이터를 안전하게 관리하는 데 탁월하지만, 방대한 양의 텍스트 데이터를 효율적으로 검색하는 데는 근본적인 한계가 있습니다. 
성능을 끌어올리기 위해 **`Full-Text Index`** 를 사용하고 **`boolean mode`** 검색을 도입하는 등 다양한 노력을 기울여도, 데이터 규모가 커질수록 검색 속도는 느려지고 복잡한 텍스트 검색은 어려워집니다.

이러한 문제에 직면한 개발자들에게 **Elasticsearch**는 새로운 해법을 제시합니다.
Elastic 사는 Elasticsearch를 **모든 형태와 규모의 데이터를 검색, 인덱싱, 분석할 수 있도록 데이터를 중앙에 저장하는 분산형 RESTful 검색 및 분석 엔진**이라고 정의합니다. 이 정의는 Elasticsearch가 RDB와 어떻게 다르고, 어떤 강점을 가졌는지 명확하게 보여줍니다.

# Elasticsearch

본 문서에서는 Elasticsearch가 검색 문제를 해결하는 과정을 따라가며 그 원리를 단계별로 살펴보겠습니다.

- **1단계: 데이터 이해** - 먼저, Elasticsearch가 데이터를 저장하고 관리하는 기본 단위인 **인덱스, 매핑, 문서, 필드**에 대해 알아봅니다. RDB와 비교하여 어떻게 데이터를 다루는지 이해하며 첫 번째 발을 내딛습니다.
- **2단계: 확장성과 안정성** - 단일 서버의 한계를 뛰어넘는 **분산형 구조**를 통해 대규모 데이터를 어떻게 처리하고 안정성을 보장하는지 설명합니다.
- **3단계: 검색의 마법** - Elasticsearch의 핵심 동작인 **분석기(Analyzer)**와 **역색인(Inverted Index)**이 어떻게 검색 속도와 정확성을 결정하는지 파악합니다.
- **4단계: 시스템과의 대화** - 마지막으로, SQL과는 다른 JSON 기반의 **RESTful API**를 통해 어떻게 시스템과 소통하는지 살펴봅니다.

# **데이터 이해**

Elasticsearch의 기본 구조를 살펴보겠습니다. 비교의 편의를 위해, 우리가 익숙한 관계형 데이터베이스(RDB)와 대응시켜 보겠습니다.

(본 글은 Elasticsearch 8.10 버전을 기준으로 작성했으며, 이후에는 Elasticsearch를 **ES**라 부르겠습니다.)

<img width="1071" height="599" alt="인덱스 이미지" src="https://github.com/user-attachments/assets/33fe1443-4c8f-4b11-9653-fe97ad38747b" />

| **Elasticsearch** | **RDB(예: MySQL)** |
| --- | --- |
| Index (인덱스) | Table (테이블) |
| Mapping (매핑) | Schema (스키마) |
| Document (문서) | row(행_가로) |
| Field (필드) | Column (열_세로) |

ES에서 데이터는 **인덱스(Index)**라는 단위에 저장됩니다. 인덱스 내부에는 여러 개의 **문서(Document)**가 존재하며, 각 문서는 여러 개의 **필드(Field)**로 구성됩니다.

ES는 전통적인 RDB와 달리 NoSQL 기반이기 때문에 고정된 스키마(Schema)가 존재하지 않습니다.
그러나, 문서의 구조와 필드 타입을 정의하기 위해 **매핑(Mapping)**을 사용합니다. 매핑은 스키마와 유사한 역할을 하지만 더 유연합니다.

예를 들어, RDB에서는 테이블에 데이터를 삽입할 때 모든 컬럼을 스키마에 맞추어 채워야 합니다.
반면, ES에서는 매핑에 정의된 필드를 모두 채우지 않아도 문서를 저장할 수 있습니다. 다만 매핑에 존재하지 않는 새로운 필드를 임의로 추가하는 것은 허용되지 않습니다.

정리하면, 인덱스는 테이블에 대응하고, 매핑은 스키마와 유사하며, 문서와 필드는 각각 행(Row)과 열(Column)에 대응한다고 이해할 수 있습니다. 이러한 구조는 RDB와 유사성을 제공하면서도 더 높은 유연성을 보장합니다.

# **확장성과 안정성**

## 왜 분산형인가?

RDB가 주로 서버의 성능(CPU, 메모리, 스토리지)을 높이는 수직 확장에 의존한다면, 엘라스틱 서치는 서버의 수를 늘려 시스템을 확장하는 수평 확장에 특화되어 있습니다. 이 분산형 구조를 통해 엘라스틱 서치는 다음과 같은 두 가지 이점을 확보합니다.

- **확장성(Scalability)**: 데이터의 양이 기하급수적으로 늘어나도 유연하게 대응할 수 있습니다. RDB가 더 높은 성능의 서버를 교체해야 한다면, 엘라스틱 서치는 서버(노드)를 클러스터에 추가하기만 하면 됩니다. 엘라스틱 서치는 자동으로 데이터를 재분배하여 늘어난 용량을 효율적으로 사용합니다. 마치 도서관에 책이 늘어날 때 새로운 서기를 추가하는 것처럼, 시스템을 중단하지 않고도 용량을 확장할 수 있습니다.
- **안정성(High Availability)**: 데이터 손실과 서비스 중단을 방지합니다. 엘라스틱 서치는 모든 데이터를 프라이머리 샤드와 그 복사본인 복제본 샤드로 나누어 관리합니다. 이 두 샤드는 절대 같은 노드에 위치하지 않습니다. 따라서 한 노드에 장애가 발생하더라도 다른 노드에 있는 복제본 샤드가 즉시 그 역할을 대신함으로써, 데이터 유실 없이 서비스를 지속할 수 있습니다

## 분산형 구조의 핵심: 클러스터, 노드, 샤드

이번에도 우리에게 익숙한 RDB와 비교하며 설명을 이어나가겠습니다.

<img width="1075" height="600" alt="클러스터 이미지" src="https://github.com/user-attachments/assets/7b98c620-c13b-4a57-baa8-81fa20f35d35" />

| **Elasticsearch** | **RDB(예: MySQL)** |
| --- | --- |
| **Cluster(클러스터)** | **Database(데이터 베이스)** |
| **Node(노드)** | **서버 (MySQL은 단일서버)** |
| **Shard(샤드)** | **Partition(파티션)** |

이러한 비교를 바탕으로, 각 구성 요소를 정리하면 다음과 같습니다.

1. **클러스터(Cluster)**: 엘라스틱 서치의 가장 큰 단위로, 여러 대의 서버(노드)가 모여 하나의 시스템처럼 동작하는 운영 시스템입니다. RDB의 **'데이터베이스'**에 대응하는 상위 개념으로 이해할 수 있습니다.
2. **노드(Node)**: 클러스터를 구성하는 개별 서버 한 대를 의미합니다. 노드는 샤드를 가지고 실제 데이터를 저장하며, 검색 및 분석 작업을 처리합니다. 
3. **샤드(Shard)**: 인덱스를 물리적으로 나눈 가장 작은 단위이자, 실제 문서가 저장되는 공간입니다. 
엘라스틱 서치는 인덱스를 샤드들로 나누고, 이 샤드들을 여러 노드에 분산시켜 병렬 처리를 통한 성능 향상을 이끌어냅니다.

여기서 중요한 점은 **노드는 하나 이상의 샤드를 가지고 있으며, 이 샤드 안에 실제 문서들이 저장된다**는 것입니다. 
인덱스는 단순히 이 문서들을 묶어주는 논리적인 이름표이며, 샤드가 물리적인 저장 공간 역할을 합니다.
따라서 여러 노드가 각자 가진 샤드를 통해 동시에 문서들을 읽고 데이터를 병렬로 처리하여, 검색 및 분석 성능을 극대화할 수 있습니다.

# **검색의 마법**

엘라스틱 서치는 앞서 살펴본 분산형 구조를 기반으로, 어떻게 빠르고 정확한 검색을 제공할까요? 
그 핵심에는 **분석기(Analyzer)**와 **역색인(Inverted Index)**이라는 두 가지 '검색의 마법'이 있습니다.

### 검색의 마법 1: 분석기(Analyzer)

엘라스틱 서치는 문서를 저장하기 전에 **분석기**라는 과정을 거칩니다. 이는 텍스트 데이터를 검색에 최적화된 형태로 가공하는 일련의 작업입니다. RDB가 텍스트를 있는 그대로 저장하는 반면, 엘라스틱 서치는 분석기를 통해 텍스트를 분해하고 정규화합니다.

분석기는 다음과 같은 과정을 거칩니다.

1. **캐릭터 필터(Character Filter)**: 텍스트에서 HTML 태그를 제거하거나 특정 문자를 변경하는 등 전처리 작업을 수행합니다.
2. **토크나이저(Tokenizer)**: 텍스트를 공백, 구두점 또는 특정 규칙에 따라 개별 단어(토큰)로 분리합니다. 예를 들어, 한국어 형태소 분석기를 사용하면 '엘라스틱서치를'을 '엘라스틱서치'와 '를'로 분리할 수 있습니다.
3. **토큰 필터(Token Filter)**: 분리된 토큰을 추가적으로 가공합니다.
    - **소문자화**: 'Apple'을 'apple'로 변환하여 대소문자를 구분하지 않는 검색을 가능하게 합니다.
    - **불용어 제거**: 'a', 'the', 'is'와 같은 의미 없는 단어를 제거하여 검색 효율을 높입니다.
    - **동의어 사전**: 동의어 사전을 정의하여 '객체지향'과  'OOP'를 같은 의미로 처리하는 등 유연한 검색을 지원합니다.

### 검색의 마법 2: 역색인(Inverted Index)

분석기를 통해 가공된 단어들은 **역색인**이라는 특별한 데이터 구조에 저장됩니다. 역색인은 엘라스틱 서치의 빠른 검색을 가능하게 하는 핵심 기술입니다.

- **색인 (Index) 📄➡️ 단어**: 일반적인 색인은 문서(페이지)를 기준으로 그 안에 포함된 단어를 찾아가는 방식입니다. 책의 목차처럼, 특정 문서가 가진 내용을 순서대로 정리하는 구조입니다.
- **역색인 (Inverted Index) 📄⬅️ 단어**: 특정 **단어**를 기준으로 해당 단어가 포함된 문서를 찾아가는 방식입니다. 책의 '찾아보기'처럼, 키워드만으로 관련 문서들을 즉시 파악할 수 있는 구조입니다. 엘라스틱서치는 이 방식을 사용합니다.

**역색인의 작동 로직**

- 문서가 저장될 때 문서의 내용을 **토큰(단어)**으로 분리합니다.
- 각 토큰이 어느 문서에 속해 있는지 목록을 만들어 역색인 테이블을 구축합니다.
- 사용자가 검색어를 입력하면, 이 테이블에서 검색어에 해당하는 토큰을 찾아 연결된 문서 ID를 즉시 반환합니다.
- **장점**
    - 전체 문서 스캔 없이 즉시 검색 가능하므로, **텍스트 검색 성능**이 매우 빠르고 효율적입니다.

## 엘라스틱서치 역색인 테이블 예시

엘라스틱서치는 아래와 같은 문서를 인덱싱할 때, 역색인 테이블을 생성합니다.

- [Doc 1]: "엘라스틱서치는 검색 엔진입니다."
- [Doc 2]: "검색 엔진은 빠른 검색을 위해 색인 구조를 사용합니다."
- [Doc 3]: "빅데이터 분석에는 엘라스틱서치와 같은 분산 데이터베이스가 필요합니다."

---

### 역색인 테이블 구조 (간단 예시)

위 문서를 토큰화하면 다음과 같은 단어들이 추출되고, 각 단어에 해당 문서 ID가 매핑됩니다.

| Term (단어) | Documents (문서 ID) |
| --- | --- |
| 엘라스틱서치 | [Doc 1, Doc 3] |
| 검색 | [Doc 1, Doc 2] |
| 엔진 | [Doc 1, Doc 2] |
| 색인 | [Doc 2] |
| 빅데이터 | [Doc 3] |
| 분산 | [Doc 3] |

---

만약 사용자가 "엘라스틱서치"라는 검색어를 입력하면, 엘라스틱서치는 다음과 같이 작동합니다.

- 역색인 테이블에서 "엘라스틱서치"라는 단어를 찾습니다.
- 해당 단어에 연결된 문서 목록인 [Doc 1, Doc 3]을 바로 확인합니다.
- 이 목록을 바탕으로 [Doc 1]과 [Doc 3]의 내용을 검색 결과로 반환합니다.

이 과정에서 엘라스틱서치는 전체 문서를 스캔할 필요 없이 단어가 포함된 문서를 직접 찾아내므로 매우 효율적입니다. 

# **시스템과의 대화**

## RESTful API

엘라스틱서치는 **JSON 기반의 RESTful API**를 통해 시스템과 소통합니다.

이는 RDB에서 SQL을 사용해 쿼리를 작성하는 방식과 달리, 웹 서비스와 대화하듯이 요청과 응답을 주고받는 방식입니다.

---

### SQL vs. JSON DSL

- **RDB**: SQL을 사용해 쿼리를 작성
- **엘라스틱서치**: JSON 기반 DSL(Domain Specific Language)을 사용

**SQL 예시**

```sql
SELECT title, summary
FROM articles
WHERE title LIKE '%엘라스틱서치%'
AND post_date > '2024-01-01';
```

**JSON DSL 예시**

```json
GET /articles/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "title": "엘라스틱서치" } }
      ],
      "filter": [
        { "range": { "post_date": { "gt": "2024-01-01" } } }
      ]
    }
  }
}
```

**장점**

- JSON 구조 덕분에 **조건을 중첩**해 직관적으로 표현 가능
- `match`, `range` 등 다양한 쿼리를 조합해
    - 전문 검색(Full-text search)
    - 필터링
    - 집계(Aggregation)를 유연하게 구현 가능

# **시스템과의 대화**

## RESTful API 개요

ES는 **JSON 기반의 RESTful API**를 통해 모든 기능을 제공합니다.

이는 전통적인 RDB에서 SQL을 사용해 쿼리를 작성하는 방식과 달리, **웹 서비스와 대화하듯이 HTTP 요청과 응답을 주고받는 구조**를 가집니다.

따라서 개발자는 특정 언어나 환경에 종속되지 않고, `curl`이나 프로그래밍 언어의 HTTP 클라이언트를 활용해 쉽게 엘라스틱서치에 접근할 수 있습니다.

---

## JSON DSL의 장점

엘라스틱서치의 JSON DSL은 단순히 데이터를 조회하는 수준을 넘어, 복잡한 검색 시나리오를 유연하게 표현할 수 있도록 설계되었습니다.

### 1. 표현력 (Expressiveness)

JSON 객체의 중첩 구조를 활용하면, 조건을 계층적으로 정의할 수 있습니다.

예를 들어 `bool` 쿼리 안에 `must`, `should`, `filter` 조건을 동시에 배치하면, 단일 쿼리로도 매우 복잡한 검색 로직을 구현할 수 있습니다.

이는 SQL에서 다중 조건을 `AND`/`OR`로 연결하는 방식보다 직관적인 구조를 제공합니다.

### 2. 유연성 (Flexibility)

엘라스틱서치는 `match`, `range`, `term`, `wildcard` 등 다양한 쿼리 타입을 지원합니다.

이들을 조합하면 다음과 같은 시나리오를 쉽게 구성할 수 있습니다.

- **전문 검색 (Full-text Search)**: 자연어 기반 검색을 지원해 사용자 입력에 맞는 문서를 찾아냄
- **조건 필터링 (Filtering)**: 날짜, 숫자 범위, 상태값 등 구조적 데이터를 효율적으로 필터링
- **집계 (Aggregation)**: 데이터 분포나 통계를 계산해 분석용 결과를 바로 제공

즉, 단순 검색을 넘어 **검색 + 필터링 + 분석**을 하나의 API 호출로 처리할 수 있습니다.

### 3. 호환성 (Interoperability)

JSON DSL은 **RESTful API 위에서 동작**하기 때문에, HTTP 요청을 전송할 수 있는 환경이라면 어디서든 사용할 수 있습니다.

웹 브라우저, 서버 애플리케이션, 클라우드 함수 등 다양한 플랫폼에서 동일한 방식으로 쿼리를 실행할 수 있어, 특정 언어나 프레임워크에 종속되지 않습니다.

---

## 마무리

관계형 데이터베이스에 익숙한 개발자라면, 엘라스틱서치는 처음엔 낯설게 느껴질 수 있습니다. 인덱스와 문서, 샤드와 클러스터 같은 개념이 SQL의 테이블이나 파티션과 닮아 있으면서도, 그 동작 방식은 완전히 다르기 때문입니다.

하지만 바로 그 지점이 엘라스틱서치의 의미입니다. 단순히 SQL 문법을 대체하려는 것이 아니라, **검색과 분석에 특화된 새로운 패러다임**을 보여주기 때문이죠. RDB가 데이터의 무결성과 일관성에 집중한다면, 엘라스틱서치는 **빠른 검색, 유연한 확장, 실시간 분석**이라는 전혀 다른 목표를 중심에 두고 설계되었습니다.

따라서 이 글에서 다룬 기본 개념들을 이해하는 것만으로도, 검색과 데이터 처리 문제를 바라보는 관점이 달라질 것입니다. 앞으로 실제 프로젝트에서 “RDB만으로는 답답하다”는 순간이 온다면, 엘라스틱서치라는 또 다른 선택지가 있다는 사실을 기억하면 됩니다.

검색은 데이터베이스의 부가 기능이 아니라, 때로는 그 자체가 시스템의 핵심이 될 수 있습니다. 엘라스틱서치는 그 가능성을 보여주는 좋은 출발점입니다.
