# 관계형 데이터베이스의 한계를 넘어: Elasticsearch로 본 검색의 진화

## 관계형 데이터베이스의 숙명, 느린 검색

**관계형 데이터베이스(RDB)** 는 오늘날 대부분의 애플리케이션에서 중심적인 역할을 수행합니다.

트랜잭션 처리, 데이터 무결성, 참조 무결성, 일관성 보장 — 이 모든 것은 RDB의 강점입니다.

그러나 데이터의 규모가 수백만 건 이상으로 늘어나고, 텍스트 기반의 비정형 데이터 검색이 많아질수록 RDB는 구조적 한계에 부딪힙니다.

즉, **RDB는 저장에는 강하지만, 검색에는 약한 구조**입니다. 이 문제를 해결하기 위해 등장한 것이 **Elasticsearch(ES)** 입니다.

## 1. Elasticsearch: 검색의 본질을 다시 보다

Elasticsearch는 데이터를 색인(indexing)하고 검색(querying)하는 방식 자체를 새롭게 정의합니다.

Elastic사는 이를 다음과 같이 정의합니다.

> “모든 형태의 데이터를 실시간으로 저장·검색·분석할 수 있는 분산형 RESTful 엔진”
> 

RDB가 **데이터 저장 중심의 시스템**이라면, Elasticsearch는 **데이터 검색 중심의 시스템**입니다.

## 2. 목차

본 문서는 **ES**가 검색 문제를 해결하는 과정을 단계적으로 따라가며, 그 핵심 원리를 **데이터 모델 → 확장 구조 → 검색 원리 → 시스템 인터페이스**의 흐름으로 설명합니다.

1. **데이터 이해(Data Model)**
    - Elasticsearch의 기본 단위인 **인덱스(Index)**, **매핑(Mapping)**, **문서(Document)**, **필드(Field)**를 다룹니다.
    - 관계형 데이터베이스와 비교하며, 데이터 저장 구조의 차이를 명확히 이해합니다.
2. **확장성과 안정성(Distributed Architecture)**
    - 단일 서버의 한계를 극복하는 **분산형 구조**를 살펴보고,
    데이터의 안정성과 고가용성을 어떻게 확보하는지 분석합니다.
3. **검색의 마법(Search Internals)**
    - 핵심 동작 원리인 **분석기(Analyzer)**와 **역색인(Inverted Index)**,
    그리고 **Docs Value**, **BM25 점수 계산 알고리즘**을 탐구합니다.
4. **시스템과의 대화(System Interaction)**
    - SQL이 아닌 JSON 기반 **DSL(Domain Specific Language)**을 통해
    Elasticsearch와 소통하는 방법을 이해합니다.

## 3. 데이터 이해: 인덱스, 매핑, 문서, 필드

**ES**의 데이터 저장 단위는 **인덱스(Index)**입니다. 하나의 인덱스 안에는 JSON 형태의 여러 개의 **문서(Document)**가 존재하며, 각 문서는 여러 **필드(Field)**로 구성됩니다.
Elasticsearch는 NoSQL 기반이므로 RDB처럼 고정된 스키마를 강제하지 않습니다. 그러나 문서 구조와 필드 타입을 정의하기 위해 **매핑(Mapping)**을 사용합니다. 

| Elasticsearch | 관계형 DB |
| --- | --- |
| 인덱스 (Index) | 테이블 (Table) |
| 문서 (Document) | 행 (Row) |
| 필드 (Field) | 열 (Column) |
| 매핑 (Mapping) | 스키마 (Schema) |

(이해를 돕기 위한 RDB와 비교한 Elasticsearch 데이터 구조)

### 매핑의 유연성과 타입 추론

RDB에서는 테이블에 데이터를 삽입할 때 모든 컬럼을 스키마에 맞추어 채워야 합니다. 반면, ES에서는 매핑에 정의된 필드를 모두 채우지 않아도 문서를 저장할 수 있습니다.
다만, 매핑에 정의되지 않은 필드를 임의로 추가하는 것은 제한됩니다.

또한 Elasticsearch는 문서가 색인될 때 **필드 타입을 자동으로 추론(Dynamic Mapping)**합니다.

예:

```json
{ "price": 1000, "title": "엘라스틱서치 입문" }
```

→ `price`는 `integer`, `title`은 `text`로 자동 설정됩니다.

자동 매핑은 편리하지만, 잘못된 타입 추론은 검색 정밀도를 떨어뜨립니다.

## 4. 확장성과 안정성: 분산 구조의 핵심

RDB가 주로 서버의 성능을 높이는 수직 확장에 의존하는 반면, ES는 서버의 수를 늘려 시스템을 확장하는 **수평 확장(Scale-out)** 에 특화되어 있습니다.

ES의 수평 확장 구조를 그림과 표로 살펴보겠습니다.

<img width="1075" height="600" alt="클러스터 이미지" src="https://github.com/user-attachments/assets/7b98c620-c13b-4a57-baa8-81fa20f35d35" />

| **Elasticsearch** | **RDB(예: MySQL)** |
| --- | --- |
| **Cluster(클러스터)** | **Database(데이터 베이스)** |
| **Node(노드)** | **서버 (MySQL은 단일서버)** |
| **Shard(샤드)** | **Partition(파티션)** |

이러한 비교를 바탕으로, 각 구성 요소를 정리하면 다음과 같습니다.

- **클러스터(Cluster)**: 여러 노드가 모인 시스템 전체 (RDB의 데이터베이스에 대응)
- **노드(Node)**: 데이터를 저장하고 검색을 수행하는 개별 서버
- **샤드(Shard)**: 데이터를 물리적으로 분할한 최소 단위

이러한 분산형 구조는 다음 두 가지 핵심 이점을 제공합니다.

1. **확장성 (Scalability)**: 노드를 추가하기만 하면 용량과 처리량을 확장 가능
2. **안정성 (High Availability)**: 프라이머리 샤드와 복제본 샤드를 분리해 장애에도 서비스 지속 가능

즉, 인덱스는 논리적 단위, 샤드는 물리적 단위이며, 분산된 샤드 덕분에 노드들의 병렬 처리를 통해 ES 는 대용량 검색도 빠르게 수행할 수 있습니다.

## 5. 검색의 마법: Analyzer와 Inverted Index

Elasticsearch의 검색 성능을 결정짓는 두 가지 핵심 요소는 **분석기(Analyzer)** 와 **역색인(Inverted Index)** 입니다.

### (1) 분석기(Analyzer)

문서를 저장하기 전, Elasticsearch는 텍스트를 분석기를 통해 토큰 단위로 분리하고 정규화합니다.

1. **캐릭터 필터(Character Filter)** – HTML 태그 제거, 특정 문자 치환
2. **토크나이저(Tokenizer)** – 공백·구두점·형태소 단위로 분리
    - 예: “엘라스틱서치를” → “엘라스틱서치”, “를”
3. **토큰 필터(Token Filter)** – 소문자 변환, 불용어 제거, 동의어 처리

이 과정을 통해 검색어가 단어의 형태나 대소문자에 구애받지 않고 일관된 방식으로**검색 최적화 형태**로 저장됩니다.

### (2) 역색인(Inverted Index)

분석된 단어들은 역색인(Inverted Index)에 저장됩니다. 이는 “단어 → 문서 목록” 구조의 테이블로 전체 문서를 스캔하지 않고도 즉시 검색이 가능하도록 합니다.

### 예시 문서

- [Doc 1]: "엘라스틱서치는 검색 엔진입니다."
- [Doc 2]: "검색 엔진은 빠른 검색을 위해 색인 구조를 사용합니다."
- [Doc 3]: "빅데이터 분석에는 엘라스틱서치와 같은 분산 데이터베이스가 필요합니다."

### 역색인 테이블

| 단어 (Term) | 문서 ID (Documents) |
| --- | --- |
| 엘라스틱서치 | [Doc 1, Doc 3] |
| 검색 | [Doc 1, Doc 2] |
| 엔진 | [Doc 1, Doc 2] |
| 색인 | [Doc 2] |
| 빅데이터 | [Doc 3] |
| 분산 | [Doc 3] |

검색어 “엘라스틱서치” 입력 시, Elasticsearch는 역색인 테이블을 참조해 [Doc 1, Doc 3]을 즉시 반환합니다.

## 6. Docs Value와 점수 계산의 원리

역색인은 “어떤 단어가 어떤 문서에 포함되었는가”만 알려줍니다. 그러나 실제 검색 결과를 사용자에게 제공하려면 문서 간 **정렬(Sorting)**, **집계(Aggregation)**, **점수(Scoring)** 계산이 필요합니다.

이때 등장하는 핵심 구조가 **Docs Value**입니다.

| 구분 | 역색인(Inverted Index) | Docs Value |
| --- | --- | --- |
| 목적 | 단어 → 문서 ID | 문서 ID → 필드 값 |
| 저장 구조 | 단어 중심(Sparse) | 열(column) 중심(Dense) |
| 주요 용도 | 검색 일치 확인 | 정렬, 집계, 통계 연산 |

Docs Value는 Lucene 레벨에서 디스크 기반으로 저장되며, 필요할 때만 메모리에 로드됩니다.

이를 통해 ES는 대규모 데이터의 정렬을 효율적으로 수행합니다.

## 7. 시스템과의 대화: JSON DSL

Elasticsearch는 SQL 대신 **JSON 기반 DSL(Domain Specific Language)**을 사용해 RESTful API로 통신합니다.

이는 RDB에서 SQL을 사용해 쿼리를 작성하는 방식과 달리, 웹 서비스와 대화하듯이 요청과 응답을 주고받는 방식입니다.

---

### SQL vs. JSON DSL

- **RDB**: SQL을 사용해 쿼리를 작성
- **엘라스틱서치**: JSON 기반 DSL(Domain Specific Language)을 사용

**SQL 예시**

```sql
SELECT title, summary
FROM articles
WHERE title LIKE '%엘라스틱서치%'
AND post_date > '2024-01-01';
```

**JSON DSL 예시**

```json
GET /articles/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "title": "엘라스틱서치" } }
      ],
      "filter": [
        { "range": { "post_date": { "gt": "2024-01-01" } } }
      ]
    }
  }
}
```

**장점**

- JSON 구조 덕분에 **조건을 중첩**해 직관적으로 표현 가능
- `match`, `range` 등 다양한 쿼리를 조합해
    - 전문 검색(Full-text search)
    - 필터링
    - 집계(Aggregation)를 유연하게 구현 가능


---

## 8. 한계와 고려사항

1. **트랜잭션 부재** – ES는 RDB의 ACID를 완전 보장하지 않습니다.
2. **데이터 중복** – 역색인 구조상 단어 단위 중복 저장이 필연적입니다.
3. **튜닝 복잡도** – 분석기 설정, 샤드 수, 캐싱 전략에 따라 성능이 달라집니다.

따라서 ES는 검색 중심 시스템에는 이상적이지만, 모든 데이터를 대체하는 RDB의 완전한 대안은 아닙니다.

## 검색은 시스템의 부가 기능이 아니다

RDB가 **데이터의 일관성**을 보장한다면, Elasticsearch는 **검색의 속도·유연성·확장성**을 제공합니다.

검색은 단순 기능이 아니라, 시스템 전체의 사용자 경험과 성능을 결정짓는 핵심 요소입니다.

“RDB만으로는 답답하다”는 순간, Elasticsearch는 개발자에게 또 다른 선택지를 제시합니다.
