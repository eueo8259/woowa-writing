# SSE 기반 실시간 동기화 구조 개선기

보따리 프로젝트에서는 여러 사용자가 동시에 협업하는 상황이 많았습니다.  
팀의 상태나 아이템 담당 정보가 바뀔 때마다 모든 참여자가 이를 실시간으로 확인할 수 있어야 했습니다.  
이를 위해 초기에는 **SSE(Server-Sent Events)** 를 사용해 서버에서 변경 알림을 push하고,  
클라이언트가 해당 알림을 받으면 **HTTP GET 요청**을 통해 전체 데이터를 다시 조회하는 구조를 사용했습니다.  
일종의 Zero-Payload 방식이었습니다.

이 방식은 단순하고 안정적이었지만, 프로젝트가 커질수록 비효율이 명확히 드러났습니다.  
SSE와 GET 요청을 각각 위한 커넥션이 필요했고, 변경이 발생할 때마다 서버는 각 요청을 처리하기 위한 추가 리소스를 사용했습니다.  
또한 화면 갱신을 위해 전체 데이터를 다시 내려야 했기 때문에, 실제로 바뀌지 않은 데이터까지 포함된 응답이 계속 전송되었습니다.  
이로 인해 불필요한 조회와 네트워크 낭비가 발생했습니다.

---

## 개선 방향을 잡기까지

이 문제를 해결하기 위해 가장 먼저 떠올린 아이디어는 단순했습니다.  
**SSE 메시지에 실제 변경된 데이터를 포함시키면, 추가 GET 요청 없이 화면을 바로 갱신할 수 있지 않을까?**

하지만 곧 여러 문제가 예상되었습니다.  
우선, 클라이언트에서 SSE 페이로드만으로 각 화면을 갱신하는 로직을 직접 구현해야 했습니다.  
현재 팀 보따리의 화면 구조는 각 화면별로 API가 분리되어 있고, 필요한 데이터도 다르기 때문에 공통화가 어려웠습니다.  
또한 SSE 특성상 메시지 유실, 중복 수신, 순서 역전 등의 이슈가 발생할 수 있었고,  
이 경우 클라이언트의 화면 상태와 실제 DB 상태가 달라질 위험이 있었습니다.

이런 고민을 정리한 끝에, 세 가지 접근 방법을 비교해보기로 했습니다.

---

## 1. 변경 트리거만 push하고, 클라이언트에서 조회 (현재 방식)

현재 구조를 그대로 유지하는 방식입니다.  
서버는 변경된 자원과 변경 종류만 클라이언트로 전달하고,  
클라이언트는 필요한 경우에만 API를 호출해 화면을 새로 그립니다.

이 방식의 장점은 단순함이었습니다.  
메시지 순서에 의존하지 않고, 일부 메시지가 누락되어도  
다음 GET 요청을 통해 최종 상태를 다시 맞출 수 있습니다.  
또한 화면마다 필요한 데이터를 직접 조회하므로,  
메시지를 화면에 맞춰 설계할 필요도 없었습니다.

반면, 변경이 잦아질수록 GET 요청이 늘어나 서버 부하가 커졌고,  
실제로 변경되지 않은 데이터까지 내려주어야 하는 비효율이 계속 남았습니다.

---

## 2. Sequence Number를 이용한 순서 보장 방식

두 번째 접근은 서버에서 각 SSE 메시지에 단조 증가하는 **seq number**를 부여하는 방식이었습니다.  
클라이언트는 seq number를 비교해 순서대로 반영하며, 누락된 메시지가 있을 경우 타임아웃 후 재요청을 수행합니다.  
이미 처리한 seq number의 메시지가 다시 오면 무시합니다.

이 방식의 장점은 명확했습니다.  
실제로 변경된 부분만 페이로드에 담을 수 있어 데이터 크기를 최소화할 수 있습니다.

하지만 현실적으로 구현 난도가 높았습니다.  
서버와 클라이언트가 모두 seq number 상태를 관리해야 했고,  
클라이언트에서는 순서 역전이나 누락을 처리하기 위한 버퍼링 로직,  
그리고 타임아웃 복구 로직까지 구현해야 했습니다.  
무엇보다 각 화면별로 다른 유형의 메시지를 반영해야 했기 때문에  
로직 수가 폭발적으로 늘어났습니다.

```text
(뷰 개수) X (자원 수) X (행위 종류)
= 8 X 5 X 4 = 160
```

즉, 약 160개의 로직이 필요했습니다.  
이 숫자는 유지보수성을 크게 떨어뜨렸습니다.

---

## 3. 멱등한 페이로드 + 자원별 버전 관리

세 번째 방법은 메시지를 **멱등하게(Idempotent)** 구성하는 것이었습니다.  
서버는 변경 시점의 버전(이벤트 발생 시각)을 포함해 메시지를 전송하고,  
클라이언트는 각 자원별로 `lastAppliedVersion`을 저장한 뒤  
이후 버전의 메시지만 반영하도록 했습니다.

이 접근의 장점은 메시지 순서나 유실에 거의 영향을 받지 않는다는 점입니다.  
재시도나 동시성 처리가 필요 없고,  
단순히 “이전보다 최신이면 반영”이라는 규칙만 지키면 일관성을 유지할 수 있습니다.

하지만 이 방식은 데이터 크기가 커졌습니다.  
변경된 부분뿐 아니라 해당 자원의 전체 상태를 계속 내려줘야 했기 때문입니다.  
또한 연관 자원이 있는 경우 JOIN이 많아지고,  
화면별로 필요한 스키마가 다르다 보니 메시지 구조를 일반화하기도 어려웠습니다.

---

## 실제 문제의 예시

예를 들어 `TeamAssignedItem`은 `TeamMember`와 `TeamAssignedItemInfo`를 함께 포함합니다.  
따라서 `TeamMember`의 이름이 바뀌면,  
`TeamAssignedItem`에도 영향을 주기 때문에 두 자원을 함께 내려야 했습니다.

```javascript
// GET /assigned-items/101
{
  "id": 101,
  "name": "버너",
  "assignees": [
    { "memberId": 42, "name": "김철수" },
    { "memberId": 77, "name": "이영희" }
  ]
}

// GET /team-members (이름 변경 후)
[
  { "id": 42, "name": "김철수(테크리더)" },
  { "id": 77, "name": "이영희" }
]
```

이처럼 연관된 자원이 많아질수록 서버에서 JOIN이 반복적으로 발생했고,  
조회 및 가공 비용이 지속적으로 증가했습니다.

또 다른 문제는 **메시지의 일반화가 불가능했다는 점**입니다.  
각 화면이 요구하는 데이터 구조가 다르기 때문에  
공통 메시지 포맷을 정의해도 실제로는 사용할 수 없었습니다.

예를 들어 9번 화면은 “물품별 팀원 현황”을,  
10번 화면은 “팀원별 담당 물품”을 보여줍니다.  
두 화면 모두 같은 자원을 사용하지만 필요한 데이터 구조가 완전히 다릅니다.

이 방식은 불필요한 데이터 전송을 유발하고,  
서버와 화면 간의 결합도를 높였습니다.

## 최종 결정

결론적으로, 두 가지 방식을 혼합한 **하이브리드 구조**로 결정했습니다.  
즉, 자원의 특성에 따라 멱등 페이로드 방식과 변경 트리거 방식을 구분했습니다.

- **JOIN이 필요 없는 자원**은 멱등 페이로드로 전송  
  (`TeamMember`, `TeamSharedItemInfo`, `TeamAssignedItemInfo`)
- **JOIN이 필요한 자원**은 변경 트리거만 전송  
  (`TeamSharedItem`, `TeamAssignedItem`)

예를 들어, `SHARED_ITEM_INFO`의 경우 전체 상태를 그대로 전송합니다.

```javascript
{
  "resource": "SHARED_ITEM_INFO",
  "event": "CREATE",
  "data": {
    "teamBottariId": 1,
    "infos": [
      { "id": 1, "name": "공용 간식" },
      { "id": 2, "name": "상비약" },
      { "id": 4, "name": "테스트물품" }
    ],
    "publishedAt": "2025-09-24T12:19:57.232206"
  },
  "publishedAt": "2025-09-24T12:19:57.272024"
}
```

반면 `SHARED_ITEM`처럼 다른 자원의 변경이 영향을 주는 경우에는,  
최소한의 정보만 포함한 트리거 메시지를 보냅니다.

```javascript
{
  "resource": "SHARED_ITEM",
  "event": "CREATE",
  "data": {
    "teamBottariId": 1,
    "publishedAt": "2025-09-24T12:19:57.232206"
  },
  "publishedAt": "2025-09-24T12:19:57.272217"
}
```

모든 메시지에는 `CREATE`, `DELETE`, `CHANGE`, `CHECK`와 같은 행위 타입을 포함해  
클라이언트가 어떤 종류의 변경인지 식별할 수 있도록 했습니다.  
클라이언트는 자신이 표시 중인 화면이 해당 자원의 변경에 영향을 받는 경우만 갱신을 수행합니다.

---

이렇게 구성한 하이브리드 전략을 통해,  
우리는 불필요한 GET 요청을 줄이면서도 실시간성을 유지할 수 있었습니다.  
JOIN이 필요한 자원에 대해서는 여전히 조회 기반으로 안정성을 확보했고,  
독립적인 자원은 멱등 페이로드 전송으로 빠른 반영이 가능해졌습니다.

결국, “하나의 정답”은 없었습니다.  
데이터의 특성에 따라 전략을 나누는 것이 가장 현실적인 선택이었습니다.
